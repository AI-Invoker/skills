# 数值设计

## 作用定位

这个 skill 用于游戏数值设计与评审，聚焦：

- 成长体系
- 资源价值
- 经济投放与消耗
- 概率与保底
- 属性、战力与模块贡献
- 平衡风险与验证方案

它适合把零散策划笔记、Excel 结论、历史规则、投放表思路，整理成可执行的数值方案或数值评审结论。

它不负责：

- 纯 Python 编程改造
- 纯 xlsx 字段设计或工作表初始化
- 纯表格导入导出流程说明

如果重点是“表怎么建、字段怎么配、xlsx 怎么流转”，应改用 `xlsx-config`。

## 什么时候调用

当用户有以下需求时，调用这个 skill：

- 设计一条新的成长线、养成线或中后期数值扩展
- 评审一个系统的资源闭环、投放和消耗是否健康
- 设计或检查概率、保底、期望值、长期体验边界
- 分析属性、战力、来源占比、模块贡献是否合理
- 统一不同资源、奖励、成本的价值坐标系
- 判断某个版本扩展会不会冲击现有经济或成长节奏
- 将多份数值文档提炼成统一方法论或结构化方案

不适用于：

- 只问 PyXLL、pandas、openpyxl、sheet 初始化或字段命名
- 只问 Excel 工作簿如何组织、如何导入导出

## 默认项目假设

除非用户另有说明，默认项目具备以下特征：

- 长线成长
- 多条并行养成线
- 强配置驱动
- Excel / xlsx 是主要策划工作台
- 需要兼顾留存、付费和版本扩展
- 强调数值可解释、可验证，而不是只看最终结果

## 核心分析维度

### 1. 统一价值

优先回答：

- 资源、奖励、货币、材料的价值锚点是什么？
- 各系统奖励能否折算到同一价值坐标？
- 成本、产出、概率、兑换路径是否能横向比较？
- 新系统接入后会不会挤压旧系统价值？

### 2. 时间节奏

优先回答：

- 玩家第 1 / 7 / 15 / 30 天分别体验到什么？
- 功能何时开放、起量、放缓、进入长期目标？
- 是否与等级、关卡、挂机、活动或开服天数节奏冲突？
- 成长是否是一条可控时间线，而不是静态终值？

### 3. 概率边界

优先回答：

- 基础概率、权重层、保底、超级保底分别是什么？
- 玩家在 10 / 50 / 100 / 200 次后的大致结果是什么？
- 是否防止极端非酋与极端超发？
- 玩家感知价值是否与展示概率和实际结果一致？

### 4. 模块化属性架构

优先回答：

- 属性是否有统一枚举或标准化表达？
- 各系统是否独立贡献属性、同时可统一汇总？
- 面板属性、战斗属性、衍生战力是否分离？
- 是否能解释“强度究竟来自哪里”？

### 5. 数据验证

优先回答：

- 是否能按模块拆解属性来源和占比？
- 是否能模拟成长速度、产出效率、长期期望？
- 是否有真实玩家样本、分阶段样本或环境样本？
- 是否能定位强度过高和存在感过低的系统？

## 标准工作流程

处理数值需求时，默认按以下顺序推进：

### 第一步：确认系统定位

说明这个系统到底服务什么：

- 留存
- 付费
- 长期深度
- 追赶机制
- 竞争场
- 奖励渠道
- 消耗口

### 第二步：定义核心循环

至少拆出：

- 入口条件
- 核心行为
- 资源流入
- 资源流出
- 输出结果
- 长期参与理由

### 第三步：拆价值结构

至少列出：

- 输入资源
- 输出资源
- 周期总价值
- 期望价值
- 兑换路径
- 通胀或低发风险

### 第四步：拆时间结构

至少列出：

- 开放时机
- 前期体验
- 中期平台期
- 后期延展
- 与日/周/赛季/开服节奏的关系

### 第五步：拆强度贡献

需要说明：

- 提供什么属性、战力或功能效用
- 影响的是面板、战斗、排行还是后台隐藏数值
- 服务差异化、追赶，还是头部长线目标

### 第六步：设计验证方式

至少说明：

- 如何验证获取是否合理
- 如何验证消耗是否成立
- 如何验证最终强度或价值是否达标

## 常见专题检查项

### 抽卡 / 招募

- 卡池分层
- 阵营/派系/品质权重
- 保底和超级保底
- 长周期期望
- 高价值目标投放控制
- 玩家感知公平与商业目标平衡

### 养成系统

- 达成里程碑的总成本
- 材料分类和稀缺度
- 历史投入能否继承、回退、转换
- 是否具备足够的长期目标

### 属性 / 战力系统

- 属性表达是否标准化
- 固定值和百分比投放逻辑
- 面板与战斗属性是否分离
- 来源拆解能力
- 战力或综合评分公式是否可解释

### 经济 / 奖励系统

- 统一价值折算
- 日/周/赛季总投放
- 稳定来源与随机来源比例
- 消耗深度
- 奖励是否匹配账号阶段

## 风险识别清单

如果出现以下情况，需要主动指出：

- 系统有奖励，但没有有效消耗口
- 系统有深度，但没有稳定获取路径
- 概率设计只有单次概率，没有长期期望边界
- 只看最终战力，不看来源构成
- 新系统增加强度，却没有说明它占整体结构的比例
- 多条养成线高度重叠，辨识度不足
- 功能开放时机与旧系统抢节奏
- 价值描述停留在感觉层，无法统一折算

## 推荐输出格式

根据任务类型，优先使用以下结构：

### A. 数值方案

1. 目标
2. 玩家阶段
3. 核心循环
4. 资源结构
5. 成长结构
6. 控制点
7. 风险
8. 验证方案

### B. 平衡评审

1. 当前设计意图
2. 主要压力点
3. 经济风险
4. 成长风险
5. 战斗/环境风险
6. 缺失验证
7. 调整建议

### C. 方法论沉淀

1. 设计目标
2. 价值锚点
3. 时间节奏
4. 概率/获取逻辑
5. 成长结构
6. 来源与消耗设计
7. 验证方式
8. 版本扩展空间

## 响应原则

- 先回答“为什么这么配”，再回答“具体配多少”
- 区分设计意图、实现机制和预期结果
- 信息不完整时，明确写出假设
- 优先使用结构化表达，而不是空泛策划话术
- 能给出验证口径时，不只给静态结论
- 如果问题本质是表格配置而非数值设计，主动建议切换到 `xlsx-config`

- “帮我设计一个中后期开启的新成长系统，并给出资源闭环”

- “帮我评审这个养成系统的投放、消耗和节奏”
- “把这些数值文档整理成统一的方法论”
- “分析战力来源拆解后，哪些模块贡献失衡”
